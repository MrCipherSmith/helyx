# PRD: Session Context Injection on CLI Session Start

## Overview

When a Claude Code CLI session restarts (after crash, manual restart, or supervisor bounce), the new process starts with a blank context window. Messages queued via MDR-001 are delivered, but Claude has no memory of prior conversation. This feature injects a prior-session context block into the first delivered message so Claude has working context immediately.

## Problem

- CLI sessions (Claude Code) maintain context only within a single process lifetime
- On restart, all in-session conversation history is lost
- The `composePrompt` / `getProjectHistory` history-loading mechanism exists but is **only called in standalone mode** — it is never exercised during CLI session delivery
- Users experience Claude asking "what are we working on?" after every restart

## Scope

**In scope:**
- Injecting context into the first queued message delivered to a freshly-started CLI session
- Two-tier content strategy: recent LLM-generated summary → raw messages fallback
- Guard against injecting on sessions that already had context delivered this process run

**Out of scope:**
- Changing standalone mode (already works via `composePrompt`)
- Generating new summaries on demand during injection (use pre-existing ones only for latency reasons; raw-message fallback handles the "no summary" case)
- Persistent injection tracking across poller restarts (intentional — fresh poller = fresh injection)

## Functional Requirements

### FR-1: Fresh-session detection
- The poller maintains an in-memory `Set<string>` called `injectedSessions`
- The key is `"${sessionId}:${clientId}"` — `clientId` is the Claude Code process identifier updated in `sessions.client_id` each time the CLI reconnects. This ensures the guard resets on every new Claude Code process, even when the session row (`id`) is reused via `adoptOrRename`.
- On each dequeue loop, before enriching a message, fetch `clientId` from the session record (via `sessionManager.get(sid)`), compute the key, and check if it is present in `injectedSessions`
- If not present: this is the first delivery to this process → attempt context injection → add key to set
- If present: skip injection, deliver normally

### FR-2: Context content — two-tier strategy

**Tier 1 (preferred): existing project summary**
- Query `memories` table for the most recent row where:
  - `project_path = session.projectPath`
  - `(chat_id = row.chat_id OR (chat_id = '' AND type = 'project_context'))` — `summary` records are stored with the real Telegram chat_id and must match per-chat; `project_context` records are always stored with `chat_id = ''` (hardcoded in `summarizeWork`) since they summarize the whole session across all chats, so a separate OR clause is required to reach them
  - `type IN ('summary', 'project_context')` — `project_context` is generated by `summarizeWork` on session exit and contains the richest structured output (`[DECISIONS]`, `[FILES]`, `[PENDING]` etc.); `summary` is generated by `trySummarize` on idle/overflow
  - `archived_at IS NULL`
- Order by `created_at DESC, CASE WHEN type = 'project_context' THEN 0 ELSE 1 END LIMIT 1` — the secondary sort key guarantees `project_context` wins over `summary` when timestamps collide (e.g. both written within the same millisecond)
- Use `content` as the context block
- Note: Tier 1 may be empty not only for first-ever conversations but also when the summarizer's quality gates (`isContentTrivial`, `isSummaryWorthSaving`) rejected saving a summary — Tier 2 handles this case

**Tier 2 (fallback): raw message history**
- Call `getProjectHistory(projectPath, chatId, 15)` from `memory/short-term.ts`
- Format each message as `<role>: <content>` on a single line (truncate at 200 chars per message)
- Join with newlines

**Skip injection entirely if:**
- Session has no `project_path` (standalone or orphaned session)
- Both tier 1 and tier 2 return empty results (first-ever conversation for this project)

### FR-3: Injection format
Prepend the following block to `enrichedContent` before the TTS note and before skill hints:

```
[Session context from prior conversation]
<context content>
[End context]

```

The block appears at the very top so Claude processes it before the user message.

### FR-4: Latency constraint
- The DB query for Tier 1 must complete before the message is delivered
- No aux LLM call during injection (to avoid adding latency to the first user message post-restart)
- If the DB query throws, log a warning and proceed without injection (fail-open)

### FR-5: Logging
- Log at `info` level when injection occurs: `{ sessionId, projectPath, tier: "summary"|"raw"|"skip", messageCount? }`
- Log at `warn` level if the DB query fails

## Technical Design

### File: `channel/poller.ts`

1. Add `private injectedSessions = new Set<string>()` to `MessageQueuePoller` (the actual class name)

2. Add module-level helper `buildContextBlock(projectPath, chatId, sql)`:
   - Tier 1: raw SQL query on `memories` table (using `this.ctx.sql`) with filters `project_path`, `chat_id`, `type IN ('summary', 'project_context')`, `archived_at IS NULL`, ordered `created_at DESC LIMIT 1`
   - Tier 2 fallback: call `getProjectHistory(projectPath, chatId, limit)` from `memory/short-term.ts` (uses its own db connection — this is fine, both hit the same DB)
   - Returns formatted context string or `null`

3. In the dequeue loop, fetch the session **once per batch** (before the `for (const row of rows)` loop) since `sid` is constant for the entire while-loop iteration. `sessionManager.get()` is a raw DB call with no in-process cache — calling it inside the per-row loop produces unnecessary round-trips:
   ```ts
   // Outside the per-row for loop:
   const session = await sessionManager.get(sid).catch(() => null);
   const injectionKey = `${sid}:${session?.clientId ?? ""}`;

   // Inside the per-row for loop, before building enrichedContent:
   let contextPrefix = "";
   if (session?.projectPath && !this.injectedSessions.has(injectionKey)) {
     this.injectedSessions.add(injectionKey);
     const block = await buildContextBlock(session.projectPath, row.chat_id, this.ctx.sql)
       .catch((err) => { channelLogger.warn({ err, sid }, "context-inject query failed"); return null; });
     if (block) contextPrefix = block + "\n\n";
   }
   const enrichedContent = `${contextPrefix}${ttsNote}${hint}${row.content}`;
   ```

### Session data availability
- Use `sessionManager.get(sid)` (from `sessions/manager.ts`) to get `projectPath` and `clientId` — no raw SQL needed
- `sessionManager` is not currently imported in `channel/poller.ts` — a new import must be added

### Dependencies
- `memory/short-term.ts` → `getProjectHistory` (not yet imported in `channel/poller.ts` — new import)
- `sessions/manager.ts` → `sessionManager` (not yet imported in `channel/poller.ts` — new import)
- `memory/db.ts` → `sql` (already available in poller context via `this.ctx.sql`)
- No new packages required

## Acceptance Criteria

1. After a CLI session restarts, the first message Claude receives includes a context block with either a prior summary or the last ≤15 messages
2. Subsequent messages in the same session process run are delivered without the context block
3. If no prior history exists for the project, no context block is added (no empty block injected)
4. If `project_path` is null on the session, no injection attempt is made
5. A failing DB query during injection does not drop or delay the message
6. The injection is logged with tier and session info

## Open Questions

1. **Message limit** — `getProjectHistory` already accepts a `limit` param and defaults to `CONFIG.SHORT_TERM_WINDOW` (env-configurable). Pass `process.env.CONTEXT_INJECT_LIMIT ?? CONFIG.SHORT_TERM_WINDOW` — no new env var strictly needed, but a dedicated one makes the intent explicit.
2. **Summary staleness** — The Tier 1 summary may be hours old. Should we add a max-age cutoff (e.g. skip summaries older than 24h and fall back to raw messages)? No existing cutoff mechanism in the memory retrieval path — would need to be added to the Tier 1 query.
