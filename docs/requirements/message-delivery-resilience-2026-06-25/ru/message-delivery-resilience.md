# PRD: Устойчивость доставки сообщений при недоступной сессии

## 1. Обзор

Два связанных пробела в доставке сообщений при рестарте/недоступности сессии Claude Code:
1. Голосовые сообщения молча теряются, если целевая сессия disconnected.
2. Сообщения, поставленные в очередь для disconnected-сессии, ждут бесконечно — нет таймаута и канала эскалации.

PRD покрывает исправление обоих пробелов.

---

## 2. Контекст

| Поле     | Значение |
|---|---|
| Продукт  | Helyx Telegram-бот |
| Модули   | `bot/media.ts`, `scripts/supervisor.ts`, `channel/poller.ts` |
| Роль     | Пользователь, отправляющий голосовые/текстовые сообщения в Claude Code сессии |
| Стек     | Bun + TypeScript, PostgreSQL (таблица `message_queue`), Telegram Bot API |

---

## 3. Описание проблемы

### 3.1 Голосовые сообщения теряются при disconnected-сессии

`handleVoice()` в `bot/media.ts` вызывает `routeMessage()` в момент прихода сообщения. Если маршрут — `disconnected`, обработчик сразу выходит с ошибкой — без транскрипции, без постановки в очередь. После рестарта сессии голосовое сообщение потеряно.

Текстовые сообщения (`text-handler.ts`) ведут себя корректно: disconnected-сессия с известным `session_id` ставит сообщение в очередь и уведомляет пользователя о доставке при рестарте.

### 3.2 Нет таймаута для сообщений в очереди

`scripts/supervisor.ts › checkStuckQueue()` детектит сообщения с `delivered = false` >5 минут, шлёт Telegram-алерт с кнопками и пытается `proj_start`. **Контент сообщений никуда не пересылается.** Если сессия не восстановится — содержимое теряется для пользователя.

---

## 4. Цели

- Ц-1: Голосовые сообщения переживают рестарт сессии (транскрибируются → ставятся в очередь → доставляются при переподключении).
- Ц-2: Сообщения, зависшие в очереди дольше настраиваемого порога (по умолчанию 10 мин), пересылаются в резервный Telegram-канал, чтобы пользователь мог их прочитать.
- Ц-3: Пересылка идемпотентна — каждое сообщение пересылается ровно один раз.
- Ц-4: Graceful degradation — если резервный канал не настроен, логируем предупреждение и пропускаем без крашей.

---

## 5. Не входит в scope

- НЦ-1: Изменение логики алертов и рестарта сессий в supervisor (существующее поведение сохраняется).
- НЦ-2: Автоматический реинжект пересланных сообщений в очередь другой сессии (пользователь решает сам, что делать).
- НЦ-3: Retry-логика для транскрипции.
- НЦ-4: Обработка `session_id = 0` (несвязанные топики) — существующий ответ с ошибкой корректен.

---

## 6. Функциональные требования

### ФТ-1 — Постановка голосового в очередь при disconnected-сессии

При приходе голосового сообщения для `disconnected`-маршрута с известным `session_id` (≠ 0):

1. Продолжить загрузку и транскрипцию (не выходить раньше).
2. При успешной транскрипции вставить строку в `message_queue` для этого `session_id` — аналогично текстовому обработчику.
3. Ответить пользователю транскрибированным текстом + «сообщение в очереди, доставлю при рестарте сессии».
4. При ошибке транскрипции — существующее сообщение об ошибке.

Объект `route` (включая `session_id`) фиксируется до начала транскрипции и не пересчитывается.

### ФТ-2 — Колонка `forwarded_at` в `message_queue`

Добавить nullable-колонку `TIMESTAMPTZ forwarded_at` в `message_queue`. По умолчанию `NULL`. Устанавливается в `NOW()` при пересылке сообщения в резервный канал. Запрос supervisor'а исключает строки с `forwarded_at IS NOT NULL`.

### ФТ-3 — Шаг пересылки в supervisor

В `checkStuckQueue()`, после существующего алерта, добавить шаг:

Для каждого зависшего сообщения (`delivered = false`, `forwarded_at IS NULL`, возраст > порога):

1. Определить резервный канал: env `JOINBOX_TOPIC_ID` (если задан) либо `SUPERVISOR_CHAT_ID` + `SUPERVISOR_TOPIC_ID`.
2. Если канал не настроен → лог `[supervisor] no fallback channel configured, skipping forward` и пропуск.
3. Отправить в Telegram:
   ```
   📬 Stuck message forwarded
   Project: <project>
   Session: #<session_id>
   From: <from_user>
   ---
   <content>
   ```
4. `UPDATE message_queue SET forwarded_at = NOW() WHERE id = <id>`.

### ФТ-4 — Настраиваемый таймаут

Env-переменная `STUCK_QUEUE_FORWARD_MINUTES` (целое число, по умолчанию `10`). Supervisor использует это значение для шага пересылки. Существующий 5-минутный порог алерта не меняется.

---

## 7. Нефункциональные требования

- НФТ-1: Никаких дополнительных DB-запросов в hot path (загрузка/транскрипция голосового). Колонка `forwarded_at` задействована только в фоновом цикле supervisor.
- НФТ-2: Миграция должна быть аддитивной (без блокировки таблицы — `ADD COLUMN DEFAULT NULL`).
- НФТ-3: Rate limits Telegram — пересылка сообщений последовательно с существующей retry/backoff-логикой supervisor'а.

---

## 8. Ограничения

- О-1: Объект `route` (включая `session_id`) фиксируется **до** вызова `enqueueForTopic` и **не пересчитывается** внутри async-задачи (текущий дизайн, менять нельзя).
- О-2: Уникальный индекс `ON CONFLICT (chat_id, message_id)` на `message_queue` применяется к голосовой очереди — использовать Telegram `message_id` как dedup-ключ.
- О-3: Supervisor запускается как отдельный процесс (`scripts/admin-daemon.ts`), env-переменные должны быть доступны там.

---

## 9. Граничные случаи

- Г-1: Транскрипция занимает долго; сессия рестартует и становится активной до её завершения → сообщение приходит в активную сессию (нет проблем — поллер подберёт сразу).
- Г-2: Несколько сообщений зависли для одной сессии → пересылать каждое отдельно.
- Г-3: Переслетанное сообщение позже доставлено (сессия поднялась после порога) → `delivered = true` ставится штатно; `forwarded_at` не меняется. Пользователь видит и уведомление в supervisor, и ответ сессии. Приемлемо.
- Г-4: `JOINBOX_TOPIC_ID` задан, но `SUPERVISOR_CHAT_ID` не задан → лог предупреждения и пропуск.
- Г-5: Транскрипция вернула `null` (пустое аудио) → не ставить в очередь; существующий ответ с ошибкой.

## 10. Критерии приёмки (Gherkin)

```gherkin
Feature: Голосовое переживает disconnected-сессию

  Scenario: Голосовое приходит, пока сессия рестартует
    Given форум-топик привязан к проекту "hypercape"
    And сессия "hypercape" имеет статус "disconnected"
    When пользователь отправляет голосовое сообщение в этот топик
    Then бот скачивает и транскрибирует аудио
    And вставляет строку в message_queue для session_id "hypercape"
    And отвечает "🎤 Transcribed: <text>\n⏳ Сессия перезапускается — доставлю автоматически"
    When сессия "hypercape" поднимается
    Then поллер доставляет поставленную в очередь транскрипцию

  Scenario: Голосовое приходит в топик без привязки (session_id = 0)
    Given форум-топик без привязки к проекту
    When пользователь отправляет голосовое сообщение
    Then бот отвечает "⚠️ Нет активной сессии для этого топика"
    And транскрипция НЕ выполняется

Feature: Пересылка зависших сообщений

  Scenario: Сообщение зависло дольше порога пересылки
    Given сообщение в message_queue с delivered = false
    And created_at 11 минут назад (порог = 10)
    And forwarded_at IS NULL
    And SUPERVISOR_CHAT_ID и SUPERVISOR_TOPIC_ID заданы
    When запускается checkStuckQueue
    Then контент сообщения публикуется в supervisor-канал
    And message_queue SET forwarded_at = NOW() для этой строки
    And на следующем цикле строка исключается из пересылки

  Scenario: Резервный канал не настроен
    Given SUPERVISOR_CHAT_ID не задан
    And сообщение зависло дольше порога
    When запускается checkStuckQueue
    Then Telegram-сообщение НЕ отправляется
    And логируется предупреждение: "[supervisor] no fallback channel configured, skipping forward"
    And forwarded_at остаётся NULL
```

## 11. Верификация

### Ручное тестирование

1. Запустить бот + admin-daemon с привязанным форум-топиком.
2. Остановить Claude Code сессию для проекта.
3. Отправить голосовое в топик — проверить: транскрипция показана, «в очереди» уведомление.
4. Запустить сессию — проверить: сообщение доставлено.
5. Снова остановить сессию. Отправить текстовое. Подождать 11 минут (или снизить порог через env).
6. Проверить: пересланное сообщение появилось в supervisor-канале. `forwarded_at` выставлен в БД.
7. Рестартовать сессию — убедиться, что сообщение всё ещё доставлено, двойной пересылки нет.

### Проверка БД

```sql
SELECT id, session_id, content, created_at, forwarded_at, delivered
FROM message_queue
WHERE forwarded_at IS NOT NULL
ORDER BY created_at DESC;
```

### Наблюдаемость

- `[supervisor] forwarded stuck message #<id> to fallback channel` — лог при успехе.
- `[supervisor] no fallback channel configured, skipping forward` — лог при отсутствии конфига.

## 13. Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Рост расходов на транскрипцию (обработка голосовых даже для мёртвых сессий) | Низкая | Подобные случаи редки; выгода перевешивает |
| Шум в supervisor-канале при массовой гибели сессий | Средняя | Дедупликация через `forwarded_at` + группировка по проекту |
| Миграция БД на живой системе | Низкая | `ADD COLUMN DEFAULT NULL` — неблокирующая операция |
| Двойная доставка (в supervisor + сессии) | Низкая | Приемлемо; сообщение в supervisor — read-only уведомление |
